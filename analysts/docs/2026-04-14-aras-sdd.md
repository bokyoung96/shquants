# ARAS System Design Document (SDD)

## 1. Intent
Build a production-leaning MVP for the Alpha Research Agent System inside `analysts/` only. The MVP must prove the ingestion → parsing → routing → analysis → wiki → signal loop while preserving clean boundaries for a later LLM-native upgrade.

## 2. Constraints
- Scope must stay inside `analysts/`.
- No reliance on new third-party packages for the core MVP path.
- Telegram access should work with Bot API-compatible polling.
- PDF parsing quality is a known risk, so the parser must be swappable.
- Every behavior must be introduced test-first.

## 3. Recommended approach
### Option A — Bot API + deterministic pipeline (**chosen**)
Use a Python package with a hexagonal architecture. Poll Telegram updates, download PDFs, run best-effort parsing, route with explicit taxonomy rules, and generate deterministic insights/wiki artifacts. This delivers a working crawl/analyze/store loop now without blocking on LLM vendor choices.

### Option B — Telethon-first crawler
Use a richer Telegram client and deeper channel scraping from day one. Rejected for the MVP because it adds dependency and auth complexity before the pipeline shape is validated.

### Option C — LLM-first prototype
Skip structure and prompt a model directly from PDFs. Rejected because it hides state, weakens testability, and makes later operational hardening harder.

## 4. Bounded MVP
### In scope
- Telegram update polling for PDF messages
- Duplicate suppression using SQLite state
- Raw PDF persistence + metadata capture
- Best-effort text extraction from simple/text-heavy PDFs
- Rule-based sector/macro routing
- Deterministic insight synthesis and coordinator merge
- Markdown wiki materialization
- Keyword repetition / sentiment delta / conflict heuristics
- CLI-based `run-once` orchestration

### Out of scope
- Real-time webhook server
- OCR for scanned PDFs
- Full chart/table extraction
- Vector database / embeddings
- Live LLM calls
- Trading bot execution

## 5. Top-level architecture
```text
Telegram Channel
    ↓
TelegramFetcher
    ↓
Raw PDF Store + SQLite Dedupe Ledger
    ↓
BestEffortPdfExtractor + DocumentParser
    ↓
TaskRouter
    ↓

    ↓
WikiBuilder
    ↓
SignalEngine
```

## 6. Module responsibilities
### `config.py`
Centralize writable paths and runtime parameters (data roots, polling limits, signal thresholds).

### `domain.py`
Own typed dataclasses for reports, parsed documents, routes, insights, and pipeline summaries.

### `storage.py`
Own the SQLite ledger for downloaded file ids, ingested document metadata, emitted insights, and signal snapshots.

### `fetcher.py`
Convert Telegram update payloads into saved local PDFs. Only accept messages with PDF media. Protect against duplicate downloads with `file_unique_id`.

### `parser.py`
Perform best-effort PDF text extraction and structure the result into sections, entities, and tickers. Extraction failures degrade gracefully and still preserve metadata.

### `router.py`
Map parsed content into one or more domain routes using explicit keyword taxonomies.

### `agents.py`
Produce deterministic analyst summaries, bull/bear cases, drivers, and risks. A coordinator consolidates multiple route outputs into a final insight bundle.

### `wiki.py`
Write markdown pages under `data/wiki/<lane>/<topic>/...` and maintain a small JSON index for lookup.

### `signal.py`
Aggregate insights into repeated-keyword summaries, sentiment drift, and cross-report conflict flags.

### `pipeline.py`
Orchestrate the entire run and return a compact run summary.

### `cli.py`
Provide `run-once`, `show-config`, and fixture-driven test/demo entrypoints.

## 8. Failure model
- **Telegram/network failure:** surface retriable error, persist last safe offset only after successful processing.
- **Duplicate PDF:** skip download and log as `skipped_duplicate`.
- **PDF parse failure:** keep metadata/raw file, store empty content, and mark parse quality as degraded.
- **Routing miss:** send to `macro/general` fallback route.
- **Wiki write failure:** fail the run loudly so downstream state is not silently partial.

## 9. Storage layout
```text
analysts/
  data/
    raw/
    processed/
    wiki/
    signals/
    state/aras.sqlite3
```

## 10. Acceptance criteria
1. A fixture-driven run can ingest at least one PDF-like Telegram message end-to-end.
2. Duplicate telegram file ids are skipped across repeated runs.
3. Parsed documents emit sections/entities/tickers.
4. Semiconductor and rates examples route correctly.
5. Wiki pages are written under deterministic paths.
6. Signal summaries detect repeated keywords and conflicting sentiment.
7. All automated tests pass under the repo venv.

## 11. Team execution split
- **Lane 1:** ingestion + storage
- **Lane 2:** parsing + routing + analysis
- **Lane 3:** wiki + signals + verification

This split minimizes overlap and mirrors the runtime pipeline.
