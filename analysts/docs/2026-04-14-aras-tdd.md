# ARAS Test-Driven Development Guide

## TDD rule
No production code in `analysts/src/` without a failing test first.

## Target test pyramid
- 70% unit tests for parser/router/agents/signal logic
- 20% integration tests for fetcher/storage and wiki emission
- 10% end-to-end pipeline tests

## Red → Green plan
### Slice 1 — Fetcher dedupe
1. Write a failing test proving only unseen PDF messages are downloaded.
2. Verify the test fails because the fetcher/store do not exist.
3. Implement the smallest Telegram client adapter + SQLite dedupe ledger.
4. Re-run the test until it passes.

### Slice 2 — Parsing structure
1. Write a failing test for pseudo-PDF extraction into title/sections/entities/tickers.
2. Verify the parser fails for the expected reason.
3. Implement the best-effort extractor and section splitter.
4. Re-run the test until it passes.

### Slice 3 — Routing
1. Write a failing test for semiconductor and rates routing.
2. Implement explicit routing taxonomies.
3. Re-run until green.

### Slice 4 — Wiki emission
1. Write a failing test for deterministic markdown page paths and index updates.
2. Implement the wiki builder.
3. Re-run until green.

### Slice 5 — Signal detection
1. Write a failing test for repeated keywords and bullish/bearish conflict detection.
2. Implement the signal engine.
3. Re-run until green.

### Slice 6 — End-to-end pipeline
1. Write a failing integration test covering fetch → parse → route → insight → wiki → signal.
2. Implement the orchestration glue only after the failure is confirmed.
3. Re-run targeted tests, then the full suite.

## Required commands
All paths below are relative to `analysts/` after `cd analysts`.
```bash
cd analysts
PYTHONPATH=src ../.venv/bin/pytest tests/test_fetcher.py -q
PYTHONPATH=src ../.venv/bin/pytest tests/test_parser.py -q
PYTHONPATH=src ../.venv/bin/pytest tests/test_router.py -q
PYTHONPATH=src ../.venv/bin/pytest tests/test_wiki.py -q
PYTHONPATH=src ../.venv/bin/pytest tests/test_signal.py -q
PYTHONPATH=src ../.venv/bin/pytest tests/test_pipeline.py -q
PYTHONPATH=src ../.venv/bin/pytest tests -q
```

## Review checkpoints
- Every test name describes one behavior.
- No mocks for pure logic when a real object is cheaper.
- Prefer fixture-based Telegram payloads over brittle mocking.
- Refactor only after a passing state exists.

## Team handoff after TDD/SDD
When `omx team` is used, workers must stay inside `analysts/` and align to the SDD lane split:
- Worker 1: fetcher + storage
- Worker 2: parser + router + agents
- Worker 3: wiki + signal + verification


## Lane 3 review findings applied
- Wiki pages and `data/wiki/index.json` must update idempotently across repeated runs.
- Signal snapshots must overwrite deterministically instead of appending duplicate summaries.
- Degraded parses must still produce explicit downstream artifacts tagged with parse-quality metadata.
